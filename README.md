# FreshKeep — 식품 유통기한 관리 시스템

> 식품 물류 현장에서 발생하는 유통기한 관리, 콜드체인 검증, 재고 격리 문제를 도메인 중심으로 설계한 마이크로서비스

---

## 프로젝트 배경
전공했던 식품공학 도메인을 바탕으로 서비스화 하는 목적에서 개발하고 있습니다.
재고 수량 관리와 더불어 **식품 안전 규칙이 코드 레벨에서 적용된 시스템**을 설계했습니다.

식품 제조 부터 유통까지 통상적인 흐름은 다음과 같습니다.
1. 제조 및 생산 : 원료 입고 > 제조 공정 > 완제품 생산
2. 자가품질검사 (의무) : 제조업체가 자체 / 외부검사기관에 의뢰
3. 자가품질검사 결과에 따른 출하 판단 : 적합(출하 승인), 부적합(출하보류, 지자체 신고 및 폐기)
4. 물류 창고 입고 : 온도 관리, 배치 관리, 유통기한 관리 시작
5. 출고 > 유통 > 소비자

- 본 프로젝트는 4번 단계에 해당합니다.
- 식품 분석 업체 경력을 녹인 프로젝트로 기획 했으나, 물류 창고 단계는 자가품질검사 적합 판정을 받은 식품을 대상이라는 것을 알았습니다.
---

## 코드 관련
1. 식품의 적정온도 구간과 입고시 식품의 온도를 소수점끼리 비교하기 위해 데이터 타입을 BigDecimal 사용했습니다.
2. 출고 요청 API : 출고 요청 하나에 여러 제품(라인)이 있을 수 있습니다.
   - 본 프로젝트의 productType은 물류 창고 내에서 구역을 결정하는 기준입니다.
    출고 요청 api에서는 1개 라인을 다루지만, 추후 라인 확장을 고려하여 lines 리스트를 추가할 계획입니다.
---

## 주요 서비스
1. 매일 자정에 재고 상태 업데이트하는 스케줄러
   - WARNING_THRESHOLD_DAYS : 유통기한일 _일 전에는 우선 출고 되어야 하는 배치
   - WARNING_THRESHOLD_PCT : (현재부터 유통기한까지 남은 날) / (전체 유통기간)  이하 이면 우선 출고 되어야 하는 배치
   - 기존 상태값이 AVAILABLE, WARNING인 재고들만 추출
   - HOLD 상태의 배치는 온도 및 품질 이슈로 막아둔거라 유통기한 경과에 따른 상태값이랑은 다름 (HOLD인 동안 유통기한이 지나면, 해제 시 EXPIRED로 상태값 업데이트)
   - 출고일 임박 관련 사항은 출고 api에서 정렬 처리
   
2. HOLD 처리한 배치 후속 작업
   - 품질관리자가 온도 측정, 냄새 확인 등 배치의 상태를 확인한 후 문제가 없을 시 HOLD를 해제 합니다.
   - 품질관리자의 판단하에 출고가 불가능한 제품은 폐기 처리합니다. (이때 배치 수량 만큼 제거)

3. 재고 현황 목록 수정 (2026.06.25)
   - 상태를 직관적으로 알수 있도록 상태별 색깔을 지정했습니다.
     AVAILABLE → 초록
     WARNING → 주황
     HOLD → 빨강
     EXPIRED/DISPOSED → 회색

4. 출고 요청
   - 특정 제품에 해당하는 배치의 모든 수량이 출고 요청 수량보다 적으면 '재고 부족으로 출고 불가' 에러를 리턴합니다. (일부만 선출고 해버리면, 부족분을 트래킹 하는 로직을 추가해야 하므로 서비스가 복잡해짐 (+ 컴플레인 대응))
